大檔案傳輸的技術核心

1. 為什麼一般網頁無法傳大檔案？

傳統網頁傳輸使用 HTTP POST，會把整個檔案讀入記憶體或一次性推送到後端伺服器。這會導致瀏覽器記憶體崩潰（Memory Out of Memory）、伺服器連線超時，或是被 Nginx 等中介代理器拒絕接收大檔案。

2. 解決方案一：前端檔案切片 (Chunking)

利用 HTML5 的 File API (Slice)。瀏覽器不一次讀取數 GB 的檔案，而是透過迴圈將檔案分割成例如 16KB ~ 1MB 的多個小碎片（Chunks），按順序依次發送，這使得傳輸數十 GB 的檔案也不會佔用過多記憶體。

3. 解決方案二：WebRTC P2P 點對點連線

本網頁所實作的技術。兩台設備之間透過 WebRTC DataChannel 連線後，檔案碎片直接從 A 設備的記憶體，經過加密通道（SCTP 協定）直達 B 設備。**全程不經過任何伺服器**，省去雲端空間成本，且私密性極高。

4. 接收大檔案時的記憶體優化

在接收端，本機如果一次把所有碎片積攢在記憶體中（ArrayBuffer），最後整合成 Blob 下載，若檔案大於 2GB 依然會引發瀏覽器崩潰。現代網頁會使用 FileSystemWritableFileStream API 或是將碎片分批暫存，以維持記憶體穩定。

💡 如何在本機測試此網頁？ 在同一台電腦上，**開啟兩個不同的瀏覽器分頁**（例如一個一般視窗，一個無痕視窗），一邊選「發起者」，另一邊選「接收者」，照著步驟複製貼上代碼，即可體驗極速的 P2P 本機傳輸！
